# Regenerate Markdown Mirrors (py-to-markdown branch upkeep)

Recurring maintenance for the `data-pipeline-tooling` repo: the `py-to-markdown`
branch holds a `.md` mirror of every `.py` source file, generated by
`scripts/generate_md_mirrors.py`. After main moves, the mirrors must be
regenerated on top of a fresh merge and pushed.

## Workflow

Run from the repo root (`/Users/nickwarters/git/data-pipeline-tooling`). The
branch is **`py-to-markdown`** — if the user names something close but wrong
(e.g. "py-to-env"), confirm you're using `py-to-markdown` and say so.

1. **Checkout + sync the branch**
   ```sh
   git checkout py-to-markdown && git pull --ff-only
   ```
2. **Merge main**
   ```sh
   git merge main
   ```
   If it merges cleanly, continue. **If there are conflicts, stop** and report
   them — do not auto-resolve; mirror content depends on the merged sources.
3. **Regenerate the mirrors** (needs the venv — system Python lacks deps):
   ```sh
   .venv/bin/python scripts/generate_md_mirrors.py
   ```
   Expect output like `Removed N existing Markdown mirror(s). / Wrote N Markdown
   mirror(s).` — the two N counts should match.
4. **Review** `git status --short`. Sanity-check that the changed/added/deleted
   `.md` files track the `.py` changes the merge brought in (a deleted `.py`
   should drop its `.md`; a new `.py` should add one).
5. **Commit + push**
   ```sh
   git add -A
   git commit -m "Regenerate Markdown mirrors after merging main"
   git push origin py-to-markdown
   ```
   The pre-commit `ruff`/`pytest` hooks will report **Skipped (no files to
   check)** because the commit is `.md`-only — that's expected, not a failure.
6. **Return to main**
   ```sh
   git checkout main
   ```

## Notes

- Commit message is always `Regenerate Markdown mirrors after merging main`
  (matches the branch's history).
- This is read-deterministic upkeep; don't edit mirrors by hand — only the
  generator writes them.
- After pushing, return to main: `git checkout main`.
