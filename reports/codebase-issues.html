1. SQLite writers overstate atomicity and can leave stale staging state

High

SqliteUpsertWriter writes a permanent staging table, performs delete/insert, commits, then drops staging without a second commit. If cleanup fails or the connection closes before persisting the drop, the next run may inherit stale objects. The comments promise atomicity, but pandas.to_sql DDL and SQLite transaction boundaries make that promise fragile.

Evidence: framework/writers.py:340, :342, :347, :355, :363, :366.

Fix: use a temporary staging table scoped to the connection, wrap delete/insert in an explicit transaction, commit cleanup deliberately, and add a failure-injection test that proves prior target rows survive.

2. Broad OperationalError swallowing can hide real data-loss conditions

High

QuarantineWriter and AccumulateByRunWriter catch any sqlite3.OperationalError around delete-by-run and treat it as “table does not exist”. That also masks schema drift, missing run_id, locked database failures after timeout, and malformed SQL. The next append can duplicate logical runs or produce confusing downstream errors.

Evidence: framework/writers.py:277-284, :425-430 and following handler.

Fix: preflight table existence with sqlite_master or catch only the exact “no such table” case; re-raise every other operational failure with target table context.

3. Several processors rely on row-wise pandas apply, which will hurt real feed sizes

Medium-high

Filter, Score, and DeriveKey execute Python callbacks per row. That is ergonomic, but the framework currently offers no batch/vectorized alternative, no warning, and no guidance about when this becomes expensive. A pipeline that works in tests can become painfully slow in production.

Evidence: framework/processors.py:74-77, :102-105, :489-497.

Fix: keep the row-callable path for simple authoring, but add vectorized processor variants or document row-count limits. Add perf-oriented tests around representative feed sizes.

4. RunRegistry ingestion rereads whole JSONL logs every run

Medium-high

The runner ingests the case-type log before and after each run, and RunRegistry.ingest() scans every line every time. Idempotency prevents duplicates, but runtime grows with total history. Operators will feel this first on long-lived case types.

Evidence: framework/run_registry.py:61-80; runner calls are in framework/runner.py around the pre-run and post-run ingest paths.

Fix: track per-log offsets or last ingested byte position, or rotate logs by date/run. Preserve idempotency, but avoid O(total log history) on every invocation.

5. QuarantineWriter fails on empty run-stamped datasets when called directly

Medium

The pipeline step only calls the reject writer when rejected rows exist, but the writer itself indexes frame["run_id"].iloc[0]. Direct use with a zero-row, correctly shaped dataset raises IndexError. That makes the writer less robust than its public Writer shape suggests.

Evidence: framework/pipeline_steps.py:216-223, framework/writers.py:270-276.

Fix: make QuarantineWriter.write() a no-op for empty frames, or require an explicit run id in its constructor if it must delete prior rejects for a zero-reject re-run.

6. The root framework package is empty while docs push facades

Medium

framework.io, framework.transform, and framework.run are clear public facades, but import framework exposes no helpful objects or pointer. New users often try package-root discovery first and get a dead end.

Evidence: framework/__init__.py:1-5, framework/io.py:1-15, framework/transform.py:1-21, framework/run.py:1-18.

Fix: either re-export the three facade modules at package root or add a small discoverability API and documentation examples that never suggest root imports.

7. CLI output drops meaningful zero values

Medium

The operator log formatter prints row metrics only when record.get(field) is truthy. A step with rows_out=0, rows_quarantined=0, or rows_excluded=0 disappears from the display, which is exactly when operators need clarity.

Evidence: pipelines/cli.py:98-110.

Fix: check is not None for metrics, matching the more careful run summary formatter at pipelines/cli.py:60.

8. Read APIs leave common CSV/Excel controls trapped behind pandas defaults

Medium

The readers intentionally hide pandas, but they expose only path/sheet/columns. Real feeds often need encoding, delimiter, dtype, date parsing, null handling, and header controls. Without a sanctioned configuration shape, users will subclass or bypass the seam.

Evidence: framework/readers.py:54-70, :76-102, :113-133.

Fix: introduce typed options objects for CSV and Excel readers. Keep pandas hidden, but expose stable domain-level knobs for common ingestion variance.

9. Comments and docstrings are carrying too much implementation history

Medium-low

The code is unusually well documented, but many comments repeat “Path keeps separators OS-agnostic”, ADR numbers, and issue IDs in places where the code is already clear. This adds scanning cost and makes maintenance harder because every behavioral change has more prose to keep aligned.

Evidence: repeated comments in framework/readers.py:62, :85, :127, :156, :208; similar comments in writers and store.

Fix: move rationale to docs/ADRs and keep inline comments for non-obvious mechanics only. Prefer a short module-level note over repeated per-constructor comments.

10. Minor dead code and formatting debt weakens signal

Low

There is at least one unused import (LAYERS in the connection factory), several long lines, and repeated manual dataclass initialization in pipeline steps. None of this is severe, but it increases noise and makes real changes harder to review.

Evidence: framework/connection.py:17; long lines in framework/processors.py:29, :344, :631; repeated object.__setattr__ blocks in framework/pipeline_steps.py:121-375.

Fix: run Ruff or equivalent linting, remove unused imports, enforce line length, and consider simple constructors over frozen dataclass subclasses where mutation is only internal stamping.

Suggested repair order

  1. Harden SQLite writer transactions and narrow OperationalError handling.
  2. Add tests for empty quarantine writes, failed upserts, and stale staging cleanup.
  3. Fix operator log zero-value display and add CLI output tests for zero rows.
  4. Decide the public import story at package root and update docs/examples.
  5. Introduce reader option objects and document supported ingestion variance.
  6. Trim repeated comments and wire up linting for small hygiene issues.
