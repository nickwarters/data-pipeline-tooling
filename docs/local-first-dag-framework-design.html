Design note · local-first data orchestration

A DAG framework where dataflow and run history are both first-class.

A proposed public API for idempotent local pipelines with step-level and pipeline-level dependencies, followed by a comparison with the current framework/ implementation.

Interface Dependency examples Concepts Current code Comparison Recommendation

Proposed Public API

The central design choice is to separate the data graph from the execution graph. A data graph says how datasets are read, transformed, validated, and written. An execution graph says which addressable run units may start, based on prior successful runs, same-day requirements, freshness windows, locks, and idempotency keys.

ResourceLocal file, SQLite table, SharePoint list, or computed dataset.
TaskPure read/process/write/validate function with declared inputs and outputs.
Run UnitA pipeline or named step with an address like pipeline_3.step_17.
ContractFreshness, same-day, success, data version, or artifact existence rule.
LedgerSQLite-backed local run history, idempotency, outputs, and decisions.
Authoring shape 
from dagpipe import (
    Pipeline, task, Source, Sink, RunBook,
    succeeded, within, same_day, days
)

pipeline = Pipeline("pipeline_3")

source = pipeline.read("read_cases", Source.csv("cases.csv"))
clean = pipeline.task("clean", clean_cases, source)
step_17 = pipeline.task(
    "step_17",
    score_cases,
    clean,
    requires=[
        succeeded("pipeline_2.step_4").within(days=7),
    ],
)
pipeline.write("publish", step_17, Sink.sqlite("gold.db", "cases"))
Execution shape 
runbook = RunBook.local(".runs")
runbook.register(pipeline)

result = runbook.run(
    "pipeline_3",
    run_date=date(2026, 6, 23),
    logical_run_id="pipeline_3:2026-06-23",
)

result.status          # "ok" | "blocked" | "failed"
result.executed_units  # step-level execution records
result.outputs         # committed artifacts

Dependency Examples

Step dependency

Step 17 in Pipeline 3 depends on Step 4 in Pipeline 2 within 7 days

step_17 = pipeline.task(
    "step_17",
    transform,
    upstream_dataset,
    requires=[
        succeeded("pipeline_2.step_4").within(days=7),
    ],
)

This requires step records to be queryable by stable address. A successful pipeline summary is not enough, because Pipeline 2 may have succeeded after Step 4 was skipped, warned, or committed an older artifact.

Pipeline dependency

Pipeline 6 requires Pipeline 4 to have succeeded on the same day

pipeline_6 = Pipeline(
    "pipeline_6",
    requires=[
        succeeded("pipeline_4").on(same_day()),
    ],
)

This is a pipeline-level freshness rule. It can be evaluated before any task runs, which keeps failures cheap and easy to explain to an operator.

General Concepts

Concept Purpose Testability hook
Task A small callable with typed inputs and outputs. It should be pure unless it is explicitly an I/O task. Run it directly with in-memory datasets and a fake context.
Address A stable identity for dependency checks: pipeline, pipeline.step, or pipeline.step.output. Dependency checks become pure queries against fake run history.
Run Context Carries execution id, logical run id, run date, base directory, registry, and clock. Inject fixed clocks, temp directories, and recording ledgers.
Ledger SQLite query store for run records, step records, outputs, locks, and orchestration decisions. Use an in-memory or temporary SQLite ledger for deterministic tests.
Idempotency Policy Defines how repeated logical runs replace, upsert, skip, or accumulate data. Assert repeated runs produce the same rows and ledger state.
Requirement A composable predicate over ledger state: success, freshness, same day, artifact present, row count, checksum. Evaluate rules without running pipelines.

Design principles

Keep

Local-first: JSONL and SQLite are enough for history, locks, and explainability.

Keep

Explicit I/O seams: readers and writers own engines, paths, credentials, and load strategies.

Keep

Idempotency by logical run id, independent from execution id.

Add

Stable step addresses as part of the public contract, not incidental log labels.

Add

A requirements DSL that can target either pipelines or steps.

Add

A planner that can dry-run and explain blocked, skipped, stale, and ready work before execution.

What The Current Code Already Has

The existing code is close in several important ways. It already has a deferred Pipeline builder with read, transform, validate, quarantine, explain, action, and write nodes. It also has a separate PipelineRunner, FreshnessRequirement, RunContext, RunLog, RunRegistry, and tools.orchestration layer.

Current dataflow builder 
p = Pipeline("cases")
r = p.read(CsvReader("cases.csv"), name="read")
c = p.task("clean", clean, r)
v = p.validate(schema, c, name="schema")
p.write(writer, v, name="write")
p.run(context)

The builder is a deferred DAG and executes leaf nodes recursively. This is flexible for fan-out and joins inside one pipeline run.

Current runner dependency 
runner.register(
    "cases",
    "selection",
    select_cases,
    freshness=(
        FreshnessRequirement("ingest", max_age_days=0),
    ),
)

Freshness is evaluated from successful pipeline run summaries in RunRegistry. It is pipeline-level, date-window oriented, and deliberately thin.

Current strengths

  • The public facades are clean: framework.core, framework.io, framework.transform, and framework.run.
  • The Dataset seam keeps pandas out of pipeline protocols and domain code.
  • Writers own load strategy, which is the right place for idempotency mechanics.
  • RunContext already separates execution id from logical run id.
  • RunLog and RunRegistry already provide local structured observability.
  • Orchestrator isolates failed scheduled items and blocks dependants while allowing independent work to continue.

Key Differences

FreshnessRequirement(upstream_pipeline, max_age_days) covers successful upstream run recency. Expand Current shape is easy to test; proposed shape scales to richer operational policies.

Pros And Cons

Proposed design

Pros

  • Can express your exact examples without workarounds.
  • Dependency logic is declarative and independently testable.
  • Step outputs can become durable, addressable artifacts.
  • Dry-run planning can give operators clear blocked/ready explanations.

Cons

  • Larger public API surface that needs stable naming rules.
  • Requires migration from incidental step names to durable step addresses.
  • More ledger schema and planner complexity.
  • Risk of turning a lightweight framework into a local Airflow clone if scope is not controlled.
Current code

Pros

  • Small, readable primitives with a clear public/internal split.
  • Good local-first operational model: files, SQLite, JSONL.
  • Strong idempotent writer strategy model already exists.
  • Pipeline-level freshness is simple and effective for many workflows.

Cons

  • Cross-pipeline dependencies are freshness-only and pipeline-level.
  • Step records are observable but not stable dependency targets.
  • Orchestration decisions and run records are split, so richer dependency queries need joining across concepts.
  • “No successful history allows first run” may be too permissive for strict production dependencies unless made configurable.

Recommended Evolution

Do not replace the current framework. Keep the existing builder, readers, writers, dataset carrier, run context, and run registry. Add a small dependency layer on top that makes step addresses and requirement predicates explicit.

Suggested next slices

Slice Change Why
1 Add RunAddress: pipeline, optional subject, optional step. Creates the vocabulary for pipeline_2.step_4 without changing execution.
2 Extend RunRegistry with step-success queries. Allows dependency checks against existing JSONL step records.
3 Generalize FreshnessRequirement into Requirement predicates. Keeps current freshness behavior while adding same-day and step-level rules.
4 Add strict first-run policy: allow, warn, or block when no upstream history exists. Production workflows often need “must have run” semantics.
5 Add a read-only planner/explain API before orchestration execution. Gives operators a deterministic answer to “what would run and why?”.

Additional Requirements: Replay, Catch-Up, And Database-Owned Schemas

These requirements change the recommendation in one important way: the framework should not only know whether a step succeeded, it should know which row attempts succeeded, failed, were quarantined, were corrected, and were replayed. That is a different level of operational fidelity from ordinary DAG orchestration.

Requirement

Quarantine bad rows, investigate manually, re-run only failed rows

The current code has the start of this: SchemaValueRulePartitioner partitions value-rule and row-check breaches into rejected rows, stamps run metadata, and writes them through a quarantine writer. The missing piece is a first-class replay contract.

Requirement

Read dated files and catch up after weekends

GlobCsvReader can read many files matching a pattern, but the framework needs a date-window file selection concept: on Monday, select Saturday, Sunday, and Monday upstream files while only running the pipeline on weekdays.

Requirement

Write into a full SQLite schema managed by migrations

This pushes the framework toward “load into existing relational tables” rather than “own all table creation.” That affects ID strategy, foreign keys, staging tables, and transaction boundaries.

Quarantine And Failed-Row Replay

Re-running only failed rows is feasible, but only if it is designed as a row-attempt lifecycle rather than an ad hoc filter over a quarantine table. Most workflow engines treat the task as the retry unit. Here, the useful retry unit is a stable source row identity.

Proposed API addition 
silver = pipeline.task(
    "silver_validate",
    coerce_and_validate,
    raw,
    quarantine=QuarantinePolicy(
        table="quarantine.case_rows",
        row_key=["source_file", "source_row_number"],
        replayable=True,
    ),
)

runbook.replay(
    "claims.silver_validate",
    where=quarantine.status.in_(["corrected", "ready"]),
    logical_run_id="claims:replay:2026-06-23",
)
Ledger shape 
quarantine_rows:
  quarantine_id
  source_row_key
  source_file
  source_row_number
  failed_rule
  original_payload_json
  corrected_payload_json
  status              -- open | corrected | replayed | waived
  first_execution_id
  replay_execution_id
  resolved_by
  resolved_at
Design point Recommendation Reason
Structural failures Still fail the step. Missing columns, unreadable files, and wrong table shape usually mean the batch contract is broken. Row-level quarantine cannot safely interpret the data.
Value failures Quarantine rows and continue when the pipeline explicitly opts in. Bad postcodes, invalid enum values, or cross-field rule breaches are often investigable row-by-row.
Replay identity Require a stable source_row_key. Without a row key, replay can duplicate rows, lose corrections, or apply fixes to the wrong source row.
Manual correction Store original payload and corrected payload separately. Operators need auditability: what arrived, what was changed, who changed it, and which replay consumed it.
Replay pipeline Replay from quarantine into the same downstream step boundary, not from the whole source. This keeps replay narrow while still using the real processors, validators, writers, and run logging.

This changes my earlier recommendation: step addresses are still needed, but row addresses become equally important for quarantine-capable steps. A future RunAddress should probably have a sibling RowAttemptAddress or ArtifactRowKey.

Dated File Catch-Up

For upstream processes that run every calendar day while local pipelines run weekdays, the source reader should be date-window aware. The schedule decides when the pipeline runs; the reader decides which source dates are due for ingestion.

Proposed reader shape 
reader = DatedFileSetReader(
    directory="/share/upstream/claims",
    pattern="claims_{date:%Y%m%d}_*.csv",
    date_range=UnprocessedDates(
        since_last_success("claims.raw_ingest"),
        through=run_date,
        include_calendar_days=True,
    ),
    columns=["claim_ref", "opened_at", "status"],
    require_all_dates=False,
)

raw = pipeline.read("read_dated_files", reader)
Concern Proposed behaviour Comparison to current code
Multiple files per day Resolve each date to zero, one, or many files; read files in deterministic sorted order. GlobCsvReader already reads many files deterministically, but it is pattern-based rather than date-window based.
Monday catch-up If the last successful source date was Friday, a Monday run can select Saturday, Sunday, and Monday files. WorkingDayCalendar governs schedule due-ness today, not source-date catch-up.
Idempotency Stamp each row with source_date, source_file, and optionally source_row_number; load by source-date or file identity. The current AccumulateByRun stamps logical run metadata. Catch-up also needs source artifact metadata.
Missing files Make strictness explicit: fail if every expected date must exist; warn/skip when upstream does not publish every day. Current GlobCsvReader fails when no file matches at all.

Destination Database With SQL Migrations

If the destination schema is defined in .sql files and migrated outside the framework, the framework should treat it as an external contract. It can validate the contract, write through explicit loading ports, and record lineage, but it should not silently reshape the database.

Proposed writer shape 
writer = RelationalWriter(
    db="/data/app.db",
    target="case",
    mode=InsertOrIgnore(
        natural_key=["source_system", "source_case_ref"],
        returning=["case_id"],
    ),
    staging_table="_stage_case",
)

case_ids = writer.write_and_return(dataset)
Contract check 
SchemaContract.from_sql_files("migrations/")
    .expect_table("case")
    .expect_unique("case", ["source_system", "source_case_ref"])
    .expect_foreign_key("case_detail", "case_id", "case.case_id")
    .verify(sqlite_connection)

Deterministic IDs vs Database-Generated IDs

My recommendation is split by table role. Use deterministic IDs for domain identity when the same real-world thing must have the same identity across re-runs, environments, and downstream joins. Use database-generated surrogate IDs inside the relational destination when the database owns the schema and has a stable natural key to de-duplicate on.

Case Prefer Why
Case identity, source entity identity, cross-run joins Deterministic ID Re-runs and replays must converge on the same identity without depending on insertion order. This matches the existing case-review direction around deterministic case_id from a natural key.
Internal database primary key with a declared unique natural key DB-generated ID is fine INSERT OR IGNORE followed by a re-read is acceptable if the natural key is unique, indexed, and used for every lookup.
Parent-child loads where children need parent IDs Either, but be explicit Deterministic parent IDs simplify child assembly. DB-generated parent IDs require a staging load, re-read mapping by natural key, then child insert in one transaction.
Manual quarantine replay Deterministic source row key Replay must target the original row attempt. A generated quarantine id is useful for UI/workflow, but not enough as the semantic row identity.

Practical rule: never rely on bare auto-increment IDs for idempotency. If the database generates IDs, pair them with a unique natural key and make the writer re-read the generated IDs by that key inside the same logical load. That pattern is testable and safe enough for SQLite.

Clarification: Task-Level Failure And Per-File Catch-Up

After revisiting the replay idea, the better design is to keep failure at the task or run-unit level. Row-level replay is possible, but it adds a lifecycle, correction workflow, row identity model, and audit surface that would dominate the framework. For this project, quarantining can remain an investigation artifact or an explicit validation mode, while normal recovery is still rerunning a failed task, step, file, or pipeline.

Multi-file catch-up should usually be orchestration

The dated-file catch-up requirement should not automatically mean “read every matching file into one stacked dataset.” That is only correct when multiple physical files together form one logical source snapshot. If each file is independently retryable, independently auditable, or independently owned by an upstream run, then orchestration should discover the files and run the same pipeline once per file or source date.

Reader concern

Many files, one logical batch

reader = GlobCsvReader(
    directory="/share/upstream/claims",
    pattern="claims_20260623_part*.csv",
)

pipeline.run(
    run_date=date(2026, 6, 23),
    logical_run_id="claims:2026-06-23",
)

Use this when the files are shards of the same business input and should succeed or fail together.

Orchestration concern

Many files, many logical runs

for source_file in discover_dated_files(
    directory="/share/upstream/claims",
    since=last_successful_source_date("claims/ingest"),
    through=run_date,
):
    runner.run(
        "claims",
        "ingest",
        base_dir,
        run_date=source_file.business_date,
        logical_run_id=f"claims:ingest:{source_file.file_id}",
        params={"source_file": source_file.path},
    )

Use this when each file should have its own run history, retry boundary, and idempotency key.

Question If yes Design choice
Are the files parts of one upstream extract? All parts must be processed together to make sense. Reader Use GlobCsvReader or a shard-aware reader that stacks them into one dataset.
Can Saturday fail while Sunday succeeds? Operators need to see and rerun only Saturday. Orchestration Run one pipeline execution per source date or file.
Does each file need its own idempotency key? Replaying one file should replace only that file's loaded rows. Orchestration Use logical_run_id derived from file identity.
Does downstream logic need a combined view? Later processing needs all successful file loads together. Both Ingest per file, then have a downstream pipeline read the accumulated raw/silver table.

Updated position

Pipeline contract: process one logical input idempotently. Orchestration contract: discover which logical inputs are due and run the pipeline for each one. Reader contract: combine files only when they are genuinely one logical input.

This fits the current codebase well. The existing ForEach orchestration helper already runs one freshly built pipeline per item with child contexts, and RunContext.logical_run_id already gives each item an idempotency key. A future source-discovery primitive could build on that rather than changing GlobCsvReader or making every pipeline internally loop over files.

User-Friendly Orchestration Schedules

Schedule rules belong in orchestration. A Pipeline should describe what work does when invoked; orchestration decides whether that Pipeline is due on a given run date. The current code already has the right low-level pieces: Weekdays, SpecificWeekdays, DayOfMonth, NthWorkingDayOfMonth, LastWorkingDayOfMonth, ManualOnly, and WorkingDayCalendar. The redesign should add an operator-friendly layer over those primitives.

Python authoring API 
ScheduledPipeline(
    "claims",
    "ingest",
    Schedule.daily(),
)

ScheduledPipeline(
    "claims",
    "quality_check",
    Schedule.on_weekdays("monday", "wednesday"),
)

ScheduledPipeline(
    "claims",
    "monthly_snapshot",
    Schedule.day_of_month(21),
)

ScheduledPipeline(
    "claims",
    "month_open",
    Schedule.nth_working_day_of_month(1),
)

Supported schedule language

Operator phrase Python Config Calendar behaviour
Daily / every working day Schedule.daily() schedule: daily Runs on working days according to WorkingDayCalendar.
Monday and Wednesday Schedule.on_weekdays("monday", "wednesday") weekdays: [monday, wednesday] Named weekdays, case-insensitive; still respects holidays if configured as working-day based.
21st of the month Schedule.day_of_month(21) day_of_month: 21 Runs when the run date's calendar day is 21; decide explicitly whether weekends should skip or run.
First working day of the month Schedule.nth_working_day_of_month(1) working_day_of_month: 1 Uses the holiday calendar; if the 1st is a weekend or holiday, moves to the first valid working day.
Last working day of the month Schedule.last_working_day_of_month() last_working_day_of_month: true Uses the holiday calendar to find the final working day in the month.
Manual only Schedule.manual_only() schedule: manual_only Never runs in automatic due-work passes.

Why not cron first?

Cron is compact, but it is not the right primary interface for this project. It does not naturally express “first working day of the month” without extra business-calendar semantics, and it is harder for operators to review safely. A friendly schedule language can still have a future cron escape hatch, but the normal path should read like the operational rule.

Separation of concerns: schedules answer “is this Pipeline due today?” Requirements answer “may this due Pipeline run now?” Source discovery answers “which files or source dates should become separate runs?” Keeping those decisions separate makes plans easier to test and explain.

Plan output should explain schedules

A friendly schedule API should produce friendly explanations. Operators should be able to see why something is ready, skipped, or blocked without reverse-engineering the schedule class.

Example plan output
