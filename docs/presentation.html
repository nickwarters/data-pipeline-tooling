Internal Framework Overview

Data Pipeline Tooling

A shared engine for every pipeline we build — standardised patterns, observable runs, queryable history, and scheduled orchestration. All backed by local SQLite and flat files; no external services required.

Pipeline Pattern

Before: Every Pipeline Is Different

  • No shared shape — every pipeline is hand-rolled
  • Validation approach varies or is absent
  • Failures are inconsistent — no standard error taxonomy
  • Impossible to query "what ran last night?" across all pipelines
Pipeline Pattern

After: One Shared Shape

  • Every pipeline follows the same medallion shape: source → raw → silver → gold
  • Swap Reader or Writer; the pipeline logic stays identical
  • Fail-fast and atomic: error aborts before the Writer is called — nothing partial lands
  • Scaffold a new feed in seconds: python -m cli scaffold orders
Pipeline Pattern

Explicit Composition, Deferred Execution

  • Tasks are wired by name — position is explicit, no hidden ordering
  • Value-rule breaches are routed to quarantine, not silently dropped
  • Error before write? Nothing lands. Warn? Flagged and continued.
  • Works identically whether the source is CSV, Excel, SQLite, or SAS
Scheduling

Built-In Schedule Definitions

  • Schedule defined once per pipeline — no cron strings scattered across configs
  • YAML overrides let environments (dev / prod) differ without code changes
  • Orchestrator decides due work for a given run date; failed items block their downstreams only
Dependent Pipelines

Freshness Requirements, Not Polling

  • Downstream pipelines declare what they need: Requirement.succeeded("ingest").same_day()
  • Guard checked before execution — no silent stale-data runs
  • Failure isolation: one set's failure blocks only its dependants; unrelated sets keep running
  • First-run behaviour is explicit: allow, warn, or block
Logging & Observability

Structured RunLog — One Record Per Step

  • Every step of every run is recorded — row counts, duration, warnings, committed flag
  • Flat JSONL: no database needed to read it; tail a file to watch a live run
  • Ingested into RunRegistry SQLite for cross-pipeline queries
Query

Queryable Run History — No External Services

  • Cross-pipeline status in one call — no log grep, no dashboard required
  • recent_row_counts() powers volume-anomaly detection (catches truncated exports)
  • runs_that_warned() surfaces schema-drift before it becomes a data error
