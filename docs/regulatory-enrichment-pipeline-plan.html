Design note · regulatory enrichment

Enrich customers from a monthly regulatory snapshot — without re-scanning it every day.

A plan for turning a huge, monthly-refreshed SAS extract into a fast, indexed lookup, and draining a per-customer enrichment backlog with one idempotent daily pipeline — using the primitives this framework already has.

The problem Shape of the answer Monthly snapshot Customer DB as queue Daily enrichment Scheduling Wiring Edge cases What to build

The problem, restated

A remote regulatory dataset is refreshed once a month. It is enormous (hundreds of GB), but compresses to a ~150 MB SAS file we hold locally. It carries the current regulatory status of every customer. We need that one data point hung onto our local customer records.

Doing the lookup live — open the extract, iterate to find the handful of customers we care about, return — is too slow to put inside a feed pipeline, and we'd pay that cost repeatedly. We want to amortise it: stand the snapshot up once per refresh as something we can probe cheaply, and let a small daily enrichment pipeline fill in the gaps.

Every silver feed already contributes the customer ids it touches into a shared customer database. So the enrichment job's input is naturally defined as "customers in that database whose regulatory data point is still missing."

The open question: enrichment should run after the day's feeds, but which feeds run varies day to day (different refresh cadences). Is a manual trigger the honest answer here, or can the framework's orchestration express this cleanly?

Short answer

Manual trigger is the fallback, not the design. Because enrichment reads its work from state (the customer DB) rather than from a fixed list of upstreams, it doesn't need to know which feeds ran today. Schedule it Weekdays() as the last item in each orchestration set, give it soft ordering dependencies (never .same_day() on every feed), and one hard freshness check on the monthly snapshot. It drains whatever backlog exists; a late feed simply gets picked up on the next sweep.

The shape of the answer: decouple via a queue-in-state

The instinct to "wait for all feeds, then enrich" couples the enrichment job to a feed set that, by your own description, isn't fixed. The framework gives us a cleaner frame: the customer DB is already an enrichment queue expressed as state. Rows with a null regulatory status are the work item list.

That turns a hard scheduling problem ("synchronise N pipelines with varying cadence") into a trivial one ("run a daily idempotent drainer"). Enrichment becomes a pure function of what's currently missing, not of what happened to run today. This is the same posture the rest of the platform takes: idempotent, re-runnable, state-driven (ADR-0004, ADR-0005).

SAS hostMonthly job emits the compressed extract; copied local (~150 MB) — ADR-0012.
regulatory subjectMonthly ingest → silver Refresh snapshot, indexed by customer_id.
customer DBEvery silver feed upserts the customer_ids it saw; null status = backlog.
enrichmentDaily: read nulls → look up indexed snapshot → upsert status back.
ConsumersSelection / Reporting read an enriched customer record. No live SAS scan.

Three pieces, three lifetimes: the snapshot turns over monthly, the queue fills continuously as feeds run, the drainer runs daily. None of them needs to know the others' schedules.

1. The monthly regulatory snapshot as a Reference Data subject

Model the extract as a Reference Data subject (per CONTEXT.md): its own per-subject medallion (regulatory/{raw,silver,gold}.db), refined by its own ingest pipeline, read-only to everything downstream. Nobody joins it live; they join its silver table.

Landing the SAS extract — ADR-0012

The remote-execution read path is exactly ADR-0012: a SasReader configured with script / copy_glob / dest runs the SAS job on the remote host, copies the compressed output back to a local landing dir, and reads the landed file through the ordinary file path — so the read works identically on Windows and macOS, and is testable end-to-end against a fixture via the StubbedRemoteRunner seam.

Build note

SasReader / RemoteRunner are specified by ADR-0012 but not yet implemented in framework/io. Until they land, the snapshot ingest can read the already-copied, decompressed extract with the existing file reader and a --param source_file=…; swapping in SasReader later is a reader substitution, not a redesign.

Why this kills the "scan it every time" cost

The point of standing the snapshot up in silver SQLite is that lookups stop being a full scan of a 150 MB file and become indexed point reads. Load it once per refresh, keyed/indexed on customer_id; thereafter every enrichment lookup is WHERE customer_id IN (…) against an index. The expensive iteration happens 12 times a year, not 250.

LayerLoad strategyContents
rawAccumulateByRunFaithful landing of each monthly extract, stamped with load_date — the system-of-record history of snapshots (ADR-0004).
silverRefreshCurrent snapshot only: one row per customer_id → regulatory_status (+ snapshot date). Schema-enforced; this is the lookup table.
gold—Not needed. This is reference data joined in Python, not a CasePool. Silver is the read surface.

Schedule: NthWorkingDayOfMonth(n) or DayOfMonth(d) to match when the upstream SAS job publishes. Add a VolumeAnomalyValidator so a truncated extract (every row valid, but half the customers silently missing) trips the run rather than quietly shrinking the snapshot.

2. The customer DB as the enrichment queue

You already plan for every silver feed to extract its customer_ids into the customer DB. Express that as a named Task in each feed's silver builder that upserts ids into a shared customers table:

In each feed's silver builder 
# after silver is assembled, register the customers it touched
seen = SelectColumns("customer_id")         # project the id
customers = store_for("customers").writer(
    SILVER, "customers",
    UpsertStrategy("customer_id"),       # merge, don't duplicate
)
p.task("register_customers", seen, silver_node)
p.write("customers", ..., customers)
customers table 
customer_id        TEXT  pk
regulatory_status  TEXT  null  # ← the gap
enriched_at        TEXT  null
snapshot_date      TEXT  null

Null regulatory_status = un-enriched. That predicate is the entire queue.

UpsertStrategy("customer_id") (already in framework.io) means a feed re-running, or two feeds sharing a customer, never duplicates the row — it merges by key. New ids arrive with a null status; enrichment fills it. ADR-0009's deterministic-key discipline keeps the id stable across runs.

3. The daily enrichment pipeline

A single ordinary pipeline at pipelines/regulatory_enrichment/, addressable by path (python -m cli run pipelines/regulatory_enrichment /data). Its run(context) is three hops:

pipelines/regulatory_enrichment/pipeline.py 
UPSTREAMS = (
    # the only HARD dependency: a current-ish monthly snapshot must exist.
    # ~40 days tolerates a late monthly refresh; it does NOT require same-day.
    Requirement.succeeded("regulatory/ingest").within_days(40),
)

def run(context):
    customers = StoreCatalog(context.base_dir).store("customers")
    regulatory = StoreCatalog(context.base_dir).store("regulatory")

    p = Pipeline("regulatory_enrichment")
    # 1. the backlog: customers still missing the data point
    missing = p.read(
        SqliteReader(customers.path(SILVER), "customers",
                    where="regulatory_status IS NULL"),
        name="missing",
    )
    # 2. look them up against the indexed snapshot (point reads, not a scan)
    snapshot = p.read(SqliteReader(regulatory.path(SILVER), "status"), name="snapshot")
    enriched = p.task("lookup", join_status, missing, snapshot)
    # 3. merge the resolved values back into the customer DB
    p.write("enrich", enriched,
            customers.writer(SILVER, "customers", UpsertStrategy("customer_id")))
    return p.run(context=context)
  • Incremental by construction. It only ever reads the null backlog, so cost scales with what's missing today, not the size of the customer base or the snapshot.
  • Idempotent. The UpsertStrategy("customer_id") write merges; re-running the same day re-resolves whatever is still null and changes nothing else. Logical run id defaults to regulatory_enrichment:<run-date> (ADR-0004), so a re-drive is safe.
  • Self-healing across a refresh. When the monthly snapshot turns over, you can additionally null-out (or re-stamp) statuses older than the new snapshot_date so the next daily sweep refreshes them — see edge cases.
  • Lookups are indexed. The join is the day's small missing set against the indexed snapshot, in Python behind the Dataset seam — never an iteration over the raw extract.

4. Scheduling — the actual question

"Run after all of today's feeds, where the feed set varies per day." The trap is reading this as a synchronisation requirement. It isn't, because enrichment doesn't consume the feeds' outputs directly — it consumes the customer DB, which the feeds have already updated by the time they finish. So we need ordering, not a guest list.

✗ The tempting-but-wrong design

Make enrichment depend_on every feed with .same_day(). Now on any day a feed isn't due (i.e. every day, by premise), that dependency is unsatisfiable and enrichment is marked blocked. You'd be fighting the orchestrator to run the very job you scheduled.

✓ The state-driven design

Schedule enrichment Weekdays() and place it last in each orchestration PipelineSet. The orchestrator runs a set's items in declared order within a pass, so enrichment runs after whichever feeds were due that day. It then drains the queue — which already reflects exactly the feeds that ran.

So: manual, or scheduled?

OptionVerdictWhy
Primary Daily Weekdays(), last in each set Recommended Matches the workload: a daily backlog drainer. Robust to the varying feed set — a late feed is simply caught on the next sweep. No manual step in the steady state.
Fallback ManualOnly() + cli run Keep available For catch-up and re-drives: a big backfill, a re-run after fixing a bad snapshot, an ad-hoc enrichment outside the daily cadence. Same pipeline, triggered by hand. Not the steady-state path.
Avoid Hard .same_day() on each feed Don't Couples enrichment to a feed set that isn't fixed; produces spurious blocked decisions every day not all feeds are due.

What about ordering dependencies — none, or soft?

Two defensible choices, both correct:

  1. No feed dependencies at all. Ordering comes purely from declaration position (last in the set). Simplest; the queue makes it safe — anything a feed missed today is drained tomorrow.
  2. Soft requirements per feed: Requirement.succeeded("<feed>/ingest").within_days(1).first_run_policy("allow"). These document intent and surface a stale feed as a warning, but with allow/warn they never block when a feed wasn't due (a None history is permitted). Use this if you want the orchestration plan to show the relationship.

Either way, the one hard dependency is the monthly snapshot: Requirement.succeeded("regulatory/ingest").within_days(40). That's a real precondition — without a current-ish snapshot there's nothing to look up — and it blocks only on a genuinely missing/stale monthly refresh, which is exactly when an operator should be told.

Settling same-day work

Run the orchestrator with --loop (run_until_complete) so a pass keeps polling until the day's due items have settled. Enrichment, being last, runs once the feeds due that day have finished — no hand-tuned ordering windows required.

5. Orchestration wiring

In the application's registry module (the one orchestrate --app imports, exposing build_pipeline_sets()):

Schedules & sets 
from tools.orchestration import (
    PipelineSet, ScheduledPipeline,
    Weekdays, DayOfMonth, NthWorkingDayOfMonth,
)
from framework.run import Requirement
from framework.run.address import RunAddress

# Reference Data set: the monthly snapshot, on its own cadence.
regulatory_set = PipelineSet("regulatory", [
    ScheduledPipeline("regulatory", "ingest",
                      NthWorkingDayOfMonth(2)),
])

# Feeds + the daily drainer, last in the set.
ingest_set = PipelineSet("feeds", [
    ScheduledPipeline("complaints_a", "ingest", Weekdays()),
    ScheduledPipeline("complaints_b", "ingest", DayOfMonth(1)),
    # ... feeds on whatever cadences they have ...
    ScheduledPipeline(
        "regulatory_enrichment", "run", Weekdays(),
        depends_on=[
            Requirement.succeeded(
                RunAddress.pipeline("ingest", subject="regulatory")
            ).within_days(40),                 # hard: snapshot must exist
        ],
    ),
])
Preview before running — Orchestrator.plan() 
2026-06-23 feeds complaints_a/ingest  ready
2026-06-23 feeds complaints_b/ingest  skipped
2026-06-23 feeds regulatory_enrich..  ready
2026-06-23 regulatory regulatory/in.. skipped

plan() reads the run registry and projects ready / skipped / blocked / already-satisfied without executing anything — so you can confirm enrichment shows ready (not blocked) on a day when only some feeds are due.

A failed feed is terminal only for its own downstream dependants in that pass; independent feeds and other sets continue. Since enrichment has no hard feed dependency, one feed failing doesn't block the day's enrichment — the failed feed's customers simply stay in the backlog until it recovers. Decisions land in <base>/_orchestration/runs.db; executions stay in RunLog / RunRegistry.

YAML overrides can flip enablement, retime a schedule, or relax freshness_days per environment without touching the Python — handy for giving production a wider snapshot window than a test box.

6. Edge cases & guardrails

SituationHandling
Monthly snapshot is late The within_days(40) hard requirement tolerates a slipped refresh; enrichment keeps draining against the still-current snapshot. Only a genuinely stale (>40d) or missing snapshot blocks it, with the reason recorded.
Snapshot refreshed — existing statuses now stale On a new snapshot_date, re-open prior rows for refresh: either null-out regulatory_status where snapshot_date < new (next sweep re-resolves them), or have the daily lookup re-resolve any row whose snapshot_date predates the current snapshot. Pick one — see open decisions.
Customer not present in the snapshot Lookup yields no match. Decide a sentinel ("unknown" / "not_in_snapshot" + enriched_at) so the row leaves the null backlog and isn't retried forever; it re-enters naturally after the next refresh if you re-open by snapshot_date.
Truncated / partial extract VolumeAnomalyValidator on the snapshot ingest trips when the row count deviates wildly from recent history — a silently-short export fails the monthly run instead of poisoning every lookup.
Big initial backfill First run faces a large null backlog. Run enrichment manually (cli run) once to drain it, or let the daily sweep chew through it — it's idempotent, so partial progress is never lost. Optionally batch the WHERE by LIMIT if a single pass is too large.
Re-drive a specific day cli run pipelines/regulatory_enrichment /data --logical-run-id … replaces that logical run's rows rather than duplicating (ADR-0004).
Catch-up after an outage The queue is durable: missed days just mean a bigger backlog next sweep. No dated-file replay needed — though --freshness-days N relaxes the snapshot window for a catch-up run.

7. What to build

  1. regulatory Reference Data subject + monthly ingest pipeline. Lands the extract (file reader now; SasReader per ADR-0012 when implemented) → silver Refresh, indexed on customer_id, with a VolumeAnomalyValidator. Schedule NthWorkingDayOfMonth/DayOfMonth.
  2. register_customers task in each silver feed builder. Projects customer_id and upserts into the shared customers table (UpsertStrategy("customer_id")). A small reusable helper so every feed wires it the same way.
  3. pipelines/regulatory_enrichment/. Read null backlog → join indexed snapshot → upsert status back. UPSTREAMS = one hard within_days(40) on regulatory/ingest.
  4. Orchestration wiring. Enrichment scheduled Weekdays(), last in the feed set(s); optional soft per-feed requirements for visibility. Verify with Orchestrator.plan() that it reads ready on a mixed-feed day.
  5. Snapshot-turnover policy (re-open by snapshot_date) — implement the chosen variant from the open decision below.
  6. Docs. New docs/regulatory-enrichment.md slice doc, a customers + regulatory Data Dictionary entry, and a CONTEXT.md note if "enrichment queue" becomes vocabulary. (Per CLAUDE.md: docs are part of the change, not a follow-up.)

Open decisions to confirm with the team

  • decide Refresh semantics on snapshot turnover: null-and-re-resolve vs. lookup-time staleness check vs. accept point-in-time staleness until a customer is touched again. Affects how "current" the data point stays between refreshes.
  • decide Soft per-feed requirements: declare them (plan-visibility) or rely purely on declaration order (simplicity)?
  • decide Where the customer DB lives: its own subject medallion (recommended, blast-radius isolation per ADR-0001) vs. folded into an existing platform store.
  • decide Snapshot window (within_days): 40 is a starting point for a monthly cadence; set it to your real tolerance for a late refresh.
The one idea to take away

Don't synchronise enrichment to a moving set of feeds. Let the customer DB be the queue, stand the monthly extract up as an indexed snapshot, and run one idempotent daily drainer last in the set. Manual trigger stays in your pocket for backfills and re-drives — but the steady state is fully scheduled, and resilient to whichever feeds happen to run on any given day.
